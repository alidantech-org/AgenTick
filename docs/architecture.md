# Skillib Architecture

## User-project boundary

Skillib discovers the Git root with `git rev-parse --show-toplevel`, then treats `<root>/agents/` as the complete project specification and runtime home. No root `AGENTS.md` or root YAML file is required.

```text
agents/
├── skillib.yml
├── skills.yml
├── skills.lock.yml
├── README.md
├── INSTRUCTIONS.md
├── rules/
├── architecture/
├── workflows/
├── patterns/
├── templates/
├── skills/          # project-owned skill sources
├── skillib/         # generated registry-managed, read-only, ignored
├── work/
└── .skillib/        # JSONL, SQLite index, reports, snapshots, ignored
```

## Runtime grouping

The CLI is intentionally shallow:

```text
packages/skillib/src/
├── cli/
│   ├── index.ts
│   └── ui.ts
└── tool/
    ├── project.ts       # Git-root discovery and configuration
    ├── templates.ts     # generated agents/ content
    ├── initialize.ts    # safe initialization and gitignore mutation
    ├── history.ts       # JSONL canonical log and SQLite query index
    ├── verify.ts        # deterministic validation
    ├── watch.ts         # filesystem observation and local dashboard
    └── runtime.ts       # public runtime surface
```

## Skills

A published skill follows the open Agent Skills shape: a directory containing `SKILL.md` with YAML frontmatter and optional `scripts/`, `references/`, and `assets/`.

- `agents/skills.yml` records user intent and version ranges.
- `agents/skills.lock.yml` records exact versions, resolved URLs, and SHA-512 integrity.
- `agents/skillib/` is regenerated by `skillib pull`, ignored by Git, and treated as read-only by humans and coding agents.
- `agents/skills/` contains project-owned source skills that may be published with `skillib push` after authentication.

Public pulls may be anonymous. Private pulls and every push require a revocable account token. The registry URL is read from the environment variable named by `agents/skillib.yml`.

## Storage boundaries

The CLI and hosted website intentionally use different persistence models.

### Local CLI runtime

JSONL is the canonical append-only local event log because it is inspectable and exportable. SQLite is maintained as a derived local index for dashboard queries. Both live under the ignored `agents/.skillib/` directory and remain offline-first and project-local.

### Hosted Next.js application

Accounts, sessions, organisations, invitations, API tokens, namespaces, skill metadata, immutable versions, analytics, and event-outbox records use PostgreSQL through Drizzle ORM. The website has no SQLite or libSQL fallback and never creates tables during requests.

Committed Drizzle migrations under `apps/site/drizzle/` are the schema source of truth. Production and CI apply migrations before starting the Next.js application. Database-backed pages are dynamically rendered so PostgreSQL is accessed at request time rather than during `next build`.

## Email and domain events

Authentication and organisation invitations use a transactional outbox:

1. the domain write and outbox event commit in one PostgreSQL transaction;
2. the dispatcher locks due events;
3. Nodemailer sends through required SMTP configuration;
4. successful deliveries are marked complete;
5. failures record an error and receive a bounded exponential retry schedule.

The protected `/api/internal/events/dispatch` endpoint allows a cron process or worker to retry pending events using `EVENT_DISPATCH_SECRET`. OTPs and invitation codes are never returned in browser API responses or rendered in the UI.
