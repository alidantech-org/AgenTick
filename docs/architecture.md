# AgenTick Architecture

## User-project boundary

AgenTick discovers the Git root with `git rev-parse --show-toplevel`, then treats `<root>/agents/` as the complete project specification and runtime home. No root `AGENTS.md` or root YAML file is required.

```text
agents/
├── agentick.yml
├── skills.yml
├── skills.lock.yml
├── README.md
├── INSTRUCTIONS.md
├── rules/
├── architecture/
├── workflows/
├── patterns/
├── templates/
├── skills/          # project-owned skill sources
├── skillib/         # generated registry-managed, read-only, ignored
├── work/
└── .agentick/       # JSONL, SQLite index, reports, snapshots, ignored
```

## Runtime grouping

The CLI is intentionally shallow:

```text
packages/agentick/src/
├── cli/
│   ├── index.ts
│   └── ui.ts
└── tool/
    ├── project.ts       # Git-root discovery and configuration
    ├── templates.ts     # generated agents/ content
    ├── initialize.ts    # safe initialization and gitignore mutation
    ├── history.ts       # JSONL canonical log and SQLite query index
    ├── verify.ts        # deterministic validation
    ├── watch.ts         # filesystem observation and local dashboard
    └── runtime.ts       # public runtime surface
```

## Skills

A published skill follows the open Agent Skills shape: a directory containing `SKILL.md` with YAML frontmatter and optional `scripts/`, `references/`, and `assets/`.

- `agents/skills.yml` records user intent and version ranges.
- `agents/skills.lock.yml` records exact versions, resolved URLs, and SHA-512 integrity.
- `agents/skillib/` is regenerated by `agentick pull`, ignored by Git, and treated as read-only by humans and coding agents.
- `agents/skills/` contains project-owned source skills that may be published with `agentick push` after authentication.

Public pulls may be anonymous. Private pulls and every push require a revocable account token. The registry URL is read from the environment variable named by `agents/agentick.yml`.

## Storage

JSONL is the canonical append-only local event log because it is inspectable and exportable. SQLite is rebuilt/maintained as a derived index for dashboard queries. The website uses a SQLite-compatible schema: local development uses a file URL; serverless production uses remote libSQL so instances share durable state.
